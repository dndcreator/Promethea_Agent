# Promethea 生产可用整改计划（基于 2026-03-23 静态评测）

更新时间：2026-03-24  
适用范围：Gateway / HTTP-WS / Conversation / Memory / Reasoning / Tool / Workflow / Security / Observability / Plugin-Channel / Computer / Config / Tests

---

## 0. 执行结论（工程视角）

这份评测报告可以直接作为整改基线，结构和拆分都正确。  
建议执行策略：

1. 先清 P0（工作流语义一致性、关键依赖缺失的统一行为）  
2. 再清关键 P1（阻塞桥接、策略体验、降级可见性）  
3. 最后做 P2/P3（异常治理、E2E 证据、文档和 smoke 自动化）

说明：当前代码已完成部分 P0/P1（工作流异步主链路、parallel 批执行、结构化依赖失败返回）。以下清单按“仍需完成”组织。

---

## 1) 启动与生命周期（4.1）

### 现状判断
- 入口统一，lifespan 初始化链路完整。
- 问题在于“失败后继续运行”路径多，能力缺失状态不够清晰。

### 必改项
1. 增加启动期能力快照（memory/reasoning/tool/workflow/mcp/plugin/channel）并持久化到状态面。
2. 初始化失败按严重度分级：`fatal`（阻断启动）/`degraded`（降级启动）/`warn`（仅告警）。
3. 去除无信息吞异常，统一落 `error_code + dependency + advice`。

### 验收标准
1. 启动后可通过一个接口看到全部关键能力状态和降级原因。
2. 关键依赖缺失时，启动日志和状态返回一致。

---

## 2) 接口层 HTTP+WS（4.2）

### 现状判断
- 路由覆盖完整，协议边界成熟。
- 失败响应缺统一“降级模板”。

### 必改项
1. 统一错误响应模型：`code/message/retryable/dependency/advice/trace_id`。
2. 服务未初始化、依赖缺失、权限不足三类错误明确区分。
3. WS 与 HTTP 的错误语义对齐（同类错误同 code）。

### 验收标准
1. 随机抽取 10 个路由，错误结构字段一致。
2. UI/CLI 能直接展示“可恢复建议”。

---

## 3) 网关控制平面（4.3）

### 现状判断
- 主功能齐全，但依赖缺失时处理分散。

### 必改项
1. 在 `gateway/server.py` 增加统一依赖检查辅助函数（tool/memory/workflow/reasoning/config/security）。
2. 统一 handler 失败路径，减少局部自定义字符串错误。
3. 对“降级继续”的路径补事件（audit + trace）。

### 验收标准
1. 网关 handler 的依赖缺失返回不再分散命名。
2. audit 可回溯一次失败请求的完整链路。

---

## 4) 对话流水线（4.4）

### 现状判断
- 六阶段设计完整。
- 主要风险是静默 fallback 导致“质量退化不可见”。

### 必改项
1. 每阶段输出 `stage_status`（ok/degraded/failed_skipped）。
2. 对 fallback 原因透出 metadata（例如 memory recall 被跳过、tool executor 不可用）。
3. response synthesis 附带统一 `capability_state` 摘要。

### 验收标准
1. 任一阶段退化时，最终响应 metadata 必有明确标记。
2. 不出现“回答正常但内部多阶段失败无痕迹”。

---

## 5) 记忆系统（4.5）

### 现状判断
- 架构能力强，部署质量敏感。

### 必改项
1. 统一后端健康探针（flat/sqlite/neo4j）并暴露 readiness。
2. 写入失败分层：可重试/不可重试/配置错误。
3. 召回与写入 gate 结果结构化回传（命中率、拒绝原因、降级状态）。

### 验收标准
1. Neo4j 不可用时系统明确降级到指定后端并可观测。
2. 记忆失败不再只留日志，API 层可看到原因。

---

## 6) 推理系统（4.6）

### 现状判断
- ReAct+ToT 完整，复杂度高。
- 问题是“跳过式降级”的可见性不足。

### 必改项
1. 对 template memory / workflow trace / moirai export 的跳过统一记录 `reason_code`。
2. 推理策略输出增加“是否走了简化路径”标识。
3. 将推理自检结果与人工确认策略分离，避免混淆。

### 验收标准
1. 推理链路所有 skip 路径均有结构化原因。
2. 回放一次推理请求可判断完整执行轨迹。

---

## 7) 工具系统（4.7）

### 现状判断
- 架构成熟，策略能力全。
- 主要是产品感知：工具可见但不可用时解释不足。

### 必改项
1. `tool_policy` 返回增强：明确“被拒绝规则命中项”。
2. 工具目录返回字段补 `callable_now`（结合策略+依赖+权限计算）。
3. MCP 缺失时返回建议安装/配置动作，而非泛化失败。

### 验收标准
1. 用户看到工具时可同时看到“现在是否可调用”。
2. 被拒绝调用可直接给出可执行修复建议。

---

## 8) 工作流系统（4.8）

### 现状判断
- 已完成关键修复：异步主链路、`parallel` 批执行、依赖失败结构化。
- 仍需补全 `graph` 语义定义与对外契约说明。

### 必改项
1. 明确 `graph` 语义（当前先按依赖图调度，非循环图）；禁止未定义语义输入。
2. 在 workflow 返回中透出 `scheduler_mode`（linear/parallel/dag-compatible）。
3. 增加 workflow 语义合同测试（声明类型 vs 实际执行顺序）。

### 验收标准
1. `workflow_type` 文档、校验、执行行为三者一致。
2. 无事件循环阻塞式 tool_step 主路径。

---

## 9) 安全边界（4.9）

### 现状判断
- 边界模型设计较好。
- 问题在于失败策略不统一（记录后继续 vs 阻断）。

### 必改项
1. 定义安全策略等级：`enforce/block`、`audit-only`、`warn`。
2. 高风险工具默认 `confirm_required`，并保留审计证据。
3. workspace 越权尝试统一错误码和审计字段。

### 验收标准
1. 安全失败路径行为一致且可追踪。
2. 高风险动作均有审计记录与执行人上下文。

---

## 10) 可观测性（4.10）

### 现状判断
- 基础较强，但缺自动诊断闭环。

### 必改项
1. 健康分级模型：`healthy/degraded/unhealthy`。
2. `doctor` 或 `ops` 增加“修复建议”字段（配置项、依赖项、命令建议）。
3. 对关键降级事件做聚合统计（近 1h/24h）。

### 验收标准
1. 运维面可直接判断是否允许放量。
2. 任一 degraded 都有明确修复建议。

---

## 11) 插件与渠道（4.11）

### 现状判断
- 插件框架完整，缺“模拟器级”系统证据。

### 必改项
1. 增加渠道适配器契约测试夹具（web/http/telegram/feishu/dingtalk/wecom）。
2. 插件加载失败返回结构化原因（manifest/entry/dependency）。
3. 渠道失败重试策略参数化。

### 验收标准
1. 每个官方渠道至少有 1 条可自动执行的连通性测试。
2. 插件加载失败能定位到具体插件与原因。

---

## 12) 本机控制（4.12）

### 现状判断
- 能力强，高权限风险高。

### 必改项
1. 明确默认权限边界（哪些命令/路径/网络可用）。
2. 对高权限操作统一审批与审计开关。
3. 工具描述文档增加“风险级别 + 前置条件”。

### 验收标准
1. 用户能在 UI/CLI 清楚看到可用/不可用能力边界。
2. 高权限调用默认不可静默执行。

---

## 13) 配置体系（4.13）

### 现状判断
- 默认配置和用户配置分层方向正确。
- 问题在于“高能力默认开 + 高依赖默认启”需要更清楚的落地提示。

### 必改项
1. 保持功能默认开启，但补“依赖未满足时的预检提示”。
2. 简单配置/高级配置严格分层，且与协议字段 1:1 对齐。
3. 新用户初始化时生成用户配置副本，并记录来源版本。

### 验收标准
1. 小白用户只看简单配置即可开始使用。
2. 高级配置修改可追溯且不污染全局默认配置。

---

## 14) 测试覆盖（4.14）

### 现状判断
- 单元/契约覆盖较广，E2E 证据不足。

### 必改项
1. 建立最小生产 smoke：`chat -> memory -> tool -> workflow -> audit`。
2. 增加“依赖缺失场景”集成测试（mcp/neo4j/workflow/tool service）。
3. 在 CI 输出 readiness 报表（通过率 + 降级统计）。

### 验收标准
1. 每次发布前自动产出 readiness 结果。
2. 关键链路的依赖缺失行为稳定可预测。

---

## 15) 优先级执行顺序（建议）

### Sprint A（必须，1-2 周）
1. Workflow 语义与契约完全一致（含 graph 语义收敛）
2. 统一依赖缺失错误模型（网关/HTTP/工具/记忆/推理）
3. 可观测性补“健康分级 + 修复建议”

### Sprint B（高优先，1 周）
1. 工具可调用性可见化（catalog + policy 解释）
2. 记忆/推理降级原因全量结构化
3. 启动生命周期能力快照

### Sprint C（稳态化，1 周）
1. 渠道/插件模拟器测试
2. 最小生产 smoke 与 CI readiness 报表
3. 文档与协议说明同步收敛
